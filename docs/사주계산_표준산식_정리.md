# 명연재 — 사주 계산 표준 산식 정리 (기준 문서)

작성일: 2026-07-01
목적: 대다수 학파가 인정하는 사주 계산 표준 산식을 정리하고, 명연재 시스템이
      이를 따르는지 코드와 대조·검증한 결과를 기록한다. 앞으로 계산 관련 작업은
      이 문서를 기준으로 삼는다.

**중요 원칙**: 이 문서는 "계산(사주 여덟 글자 세우기)"의 기준이다.
"해석·통변"은 연재 선생님 검수 영역이며 이 문서의 범위가 아니다.
계산 엔진(useResultSaju, /api/lunar, lib/saju/ganji)은 이미 검증되었으므로
새로 만들지 않고, 아래 기준대로 유지한다.

---

## 0. 두 개의 층 (핵심 이해)

사주 계산은 두 층으로 나뉜다.

1. **역법 변환** (음↔양 변환, 절기·삭망 날짜 확정)
   - 천문학의 영역. 학파와 무관.
   - 한국은 한국천문연구원(KASI)이 국가 표준으로 산출.
   - 명연재는 KASI OpenAPI를 사용 → 이 층은 자동으로 국가 표준을 따른다.

2. **간지 배정** (연·월·일·시주에 천간지지 붙이기)
   - 고전 이래 거의 이견 없는 공식들.
   - 아래 1~4에 정리. 우리 시스템은 모두 표준 준수 확인됨.

---

## 1. 연주(年柱) — 입춘 기준

- 한 해의 시작은 양력 1월 1일도 음력 설날도 아닌 **입춘(立春) 절입 순간**이다.
- 설을 쇠었어도 입춘 전이면 묵은해 간지, 연도가 바뀌었어도 입춘 전이면 전년 간지.
- 근거: 한국민족문화대백과("입춘을 기점으로 새해가 시작"), 정해만세력 등 다수 일치.
- **우리 시스템**: KASI 절기 API(getYearGanji)가 처리 → ✅ 표준 준수.

## 2. 월주(月柱) — 절기 기준 + 년상기월법(월두법)

- 월주는 음력 몇 월이냐가 아니라 **12절기(節)**로 바뀐다.
  (입춘·경칩·청명·입하·망종·소서·입추·백로·한로·입동·대설·소한)
- 24절기 중 '절(節)'만 사용, '중기(氣)'는 사용 안 함.
- 지지는 고정: 입춘~ 寅월, 경칩~ 卯월 … 순서.
- 월간은 년상기월법(월두법): 연간에 따라 정월(寅월)의 천간이 정해짐.
  - 甲己년 → 丙寅월부터,  乙庚년 → 戊寅,  丙辛년 → 庚寅,
    丁壬년 → 壬寅,  戊癸년 → 甲寅부터.
- 주의: 오픈소스 만세력에서도 "월주를 음력 월초 기준으로 계산하면 약 18%가 틀린다,
  반드시 절기 기준이어야 한다"고 지적할 만큼 이 절기 기준이 핵심이다.
- **우리 시스템**: KASI 절기 API(getMonthGanji)가 처리 → ✅ 표준 준수.

## 3. 일주(日柱) — 만세력 그대로

- 절기와 무관하게 그날의 간지를 그대로 사용.
- **우리 시스템**: getDayGanji가 양력 날짜로 계산 → ✅ 표준 준수.

## 4. 시주(時柱) — 오자둔(五子遁) 시두법 [코드 검증 완료]

- 시간(時干)은 일간(日干)에 따라 정해짐(시두법). 자시(子時) 시작 천간:
  - 甲己일 → 甲子시부터,  乙庚일 → 丙子,  丙辛일 → 戊子,
    丁壬일 → 庚子,  戊癸일 → 壬子부터.
- **우리 시스템**: useResultSaju.ts 의 calcHourPillar (groupBase=[0,2,4,6,8,0,2,4,6,8] 방식).
- **검증 결과**: 10개 일간 × 12개 시지 = 120건 전수 대조, 표준 오자둔 시두법과 100% 일치.
  - 문헌 사례도 일치: 乙庚일 卯시→己卯 / 丙辛일 寅시→庚寅 / 戊癸일 戌시→壬戌.
  - → ✅ 표준 준수 확인 완료. 계산식 변경 금지.

## 5. 윤달(閏月) 처리 — 절기로 자동 결정 [코드 확정]

### 핵심 결론
"윤5월을 5월로 보나 6월로 보나"라는 고민은 불필요하다.
월주는 음력 월 이름이 아니라 **절기**로만 정해지기 때문이다.
윤달의 역할은 오직 하나 — **정확한 양력 날짜를 찾는 것**뿐이다.
양력 날짜가 정해지면 그날이 어느 절기 구간인지로 월주가 자동 결정된다.

### 우리 시스템의 실제 처리 (app/api/lunar/route.ts 확인)
- 음력 입력일 때만 윤달 사용.
- leapMonth === "1" 이면 KASI 음→양 변환 API(getSolCalInfo)에
  `lunLeapmonth=윤` 을 넘겨 정확한 양력 날짜(solYear/solMonth/solDay)를 받는다.
- 그 양력 날짜로 연·월·일 간지를 계산.
- → 윤5월 20일과 평5월 20일은 KASI가 서로 다른 양력 날짜를 반환하고,
    그 날짜로 월주가 절기 기준으로 자동 결정됨. **계산은 표준대로 이미 정확.**

### 현재의 유일한 결함 (계산이 아니라 입력)
- 입력 UI(홈 AiManseryeokSection, welcome, newborn) 어디에도 윤달 선택 칸이 없다.
- profiles(DB)에도 윤달 컬럼이 없다.
- 그래서 항상 leapMonth='0'(평달)로만 흘러가, 윤달생은 사주가 조용히 틀린다.
- **조치 방향**: 계산식은 절대 손대지 않는다. 입력·전달·저장만 채우면 표준 정확도 달성.
  (아래 7. personKey/입력 표준화 작업에 포함)

### 향후 점검(별건)
- /api/lunar 는 KASI 호출 실패·빈 응답 방어가 약함(빈 값이면 NaN 가능).
  산식 정확도와는 별개. 오픈 전 안정화 항목으로만 기록.

## 6. 유파가 갈리는 지점 — 명연재의 채택 기준

세 가지 논쟁점이 있으나, 대다수 프로그램이 택하는 "만세력·표준시 그대로" 방식을 채택한다.

### ① 야자시 / 조자시
- 밤 23:30~01:30 자시를 둘로 나눠 일주를 다르게 볼지의 논쟁.
- 실무 다수 결론: "그냥 만세력대로 본다"(자시면 그날 일간 기준으로 단순 처리).
- **명연재 채택**: 자시(idx 0)를 그날 일간 기준으로 단순 처리. 야자시 세분화 안 함. → 다수 표준.

### ② 표준시(태양시) 보정 (동경 135° → 약 -30분)
- 시주만 별도 보정하는 방식은 모든 학파 공통이 아니며 논리적 일관성 비판도 있음.
- **명연재 채택**: 보정 안 함(표준시 그대로). → 다수 표준.
  (원하면 나중에 '선택 기능'으로 열어둘 여지는 있음.)

### ③ 태어난 시 모름
- 시주 계산 불가 → 연·월·일 세 기둥으로 해석.
- **명연재 채택**: hourIdx=null이면 시주를 '?'로 두고 세 기둥만 사용. → 표준.

## 7. 내부 식별키(personKey) 및 입력 표준 규격

계산과 별개로, "같은 사람인지" 판별하는 내부 식별키(personKey)와
사주 입력값의 내부 표현을 표준화한다. (개명이 성씨를 못 찾는 버그의 근본 해결)

### 표준 info 형태 (모든 화면 공통)
    { calType: '양력'|'음력',
      year, month, day : 문자열,
      leapMonth : '0'(평달) | '1'(윤달),
      hour : '0'~'11'(시주 인덱스 문자열) | '모름',
      gender : '남' | '여' }

### 시주 인덱스 매핑 (절대 변경 금지)
    0=子(23~01) 1=丑(01~03) 2=寅(03~05) 3=卯(05~07) 4=辰(07~09) 5=巳(09~11)
    6=午(11~13) 7=未(13~15) 8=申(15~17) 9=酉(17~19) 10=戌(19~21) 11=亥(21~23)

### 표준 personKey 규칙 (유일한 규칙)
    [calType, year, month, day, leapMonth, hourToken, gender].join('|')
    - hourToken: hour가 '모름'/null/'' → 'x', 그 외 숫자문자열 그대로
    - 모든 필드 문자열로 정규화(숫자/문자 혼용 금지)

### 발견된 버그 (검증으로 재현 완료)
- 홈(AiManseryeokSection)이 "시간 모름"을 '-1'로 저장.
- personKey 계산은 '모름'만 'x'로 변환 → '-1'은 못 알아봄.
- 결과: "시간 모름 + 홈 입력 + 비로그인/폴백" 조합에서 diagnosis 저장키(…|x|…)와
  newname 조회키(…|-1|…)가 어긋나 개명이 성씨를 못 찾음.
- 5개 사례 검증 시 '시간 모름' 2건에서 불일치 재현됨.

### 표준화 조치 (검증으로 해결 확인)
1. "시간 모름"의 내부 표현을 '모름'으로 단일화(홈의 '-1' 제거).
2. leapMonth를 '0'/'1' 정식 필드로 승격(입력·전달·저장·personKey 모두 포함).
3. 공통 헬퍼 lib/saju/myInfo.ts 하나로 info 생성·personKey를 단일화
   (profiles/URL/myinfo/직접입력 → 표준 info 변환 함수 + stdPersonKey + hourToIdx).
- 표준화 후 7개 사례(시간 모름·윤달 포함) 전부 세 경로 personKey 완전 일치 확인.
- 윤5월 vs 평5월도 서로 다른 personKey로 정상 구분 확인.

## 8. 저장 위치 원칙 (myinfo 통일 방향)

- **내 사주**(로그인한 나): profiles(DB)에 영구 저장. 로그인 방식(email/소셜) 무관.
- **남의 사주**(궁합 상대·신생아 등): URL 파라미터 또는 sessionStorage(임시).
- **localStorage 사용 금지**: 공용기기에 앞사람 정보가 남는 원인.
- 로그인 통일(email→향후 소셜)과 무관하게, 원본이 계정(profiles)에 있으면 안전.

---

## 9. 검증 자산 (재현 가능)

아래 스크립트로 언제든 재검증 가능(Node 실행).
- personKey 현재상태 검증: '시간 모름' 2건 불일치 재현.
- personKey 표준화후 검증: 7사례 전부 일치 + 윤달 구분 정상.
- 시두법 검증: 120건 전수 표준 일치.
