# 명연재 — 공통 해설 톤·용어 원칙

작성일: 2026-07-01
적용 범위: 명연재의 **모든 AI 해설** — 사주 기본 분석, 이름 풀이, 궁합,
           출산·결혼 택일, 오늘의 운세 등 사용자에게 보여주는 모든 통변 텍스트.
목적: 화면마다 프롬프트가 흩어져 있어도 해설의 톤과 용어 수준을 하나로 통일한다.
      각 화면 프롬프트는 이 문서의 규칙을 공통 전제로 포함한다.

**중요:** 이 원칙은 "사용자에게 보여주는 문장"에만 적용된다.
계산(사주 세우기·점수)은 기존 기준(docs/사주계산_표준산식_정리.md)을 그대로 따르며 절대 바꾸지 않는다.
쉬운 말은 결과 표현의 문제일 뿐, 계산 정확도와 무관하다.

---

## 0. 두 개의 대원칙

명연재 사용자(특히 젊은 층)가 사주를 보는 이유는 예언·판정이 아니라
**① 따뜻한 위로와 공감, ② 쉬운 말로 이해되는 설명**이다. 조사로도 확인됨.
모든 해설은 이 둘을 최우선으로 한다.

---

## 1. 따뜻한 위로 (톤)

- 겁주지 않는다. 불행·이별·질병·파탄을 단정하지 않는다.
- 부족한 부분도 "보완하면 더 좋아진다"는 성장의 언어로 푼다.
- 판정("이렇다")보다 조언("이렇게 하면 더 좋아요")의 태도.
- 존댓말, 다정하고 담백하게. 과장된 감탄사 남발 금지.
- 반드시 따뜻하게 마무리한다.

## 2. 쉬운 말 (용어) — "둘째 방식" 확정

**쉬운 설명을 앞세우고, 핵심 명리 용어만 괄호로 살짝 곁들인다.**
- 좋은 예: "서로 자연스럽게 끌리는 기운(일지 합)이 있어요. 상대가 나에게 필요한
  힘(용신)을 채워주는 관계라, 함께 있을수록 편안해지실 거예요."
- 나쁜 예(용어 그대로): "일지가 육합이고 용신이 상생하며 식신이 왕합니다."
- 원칙:
  - 쉬운 말이 문장의 주인. 용어는 괄호 보조로만.
  - 한 문장에 용어를 두 개 이상 넣지 않는다(괄호 남발 금지).
  - 정말 무서운 용어(원진살·투합·백호·괴강 등)는 괄호로도 노출하지 않는다.
    → 뜻만 쉬운 말로 풀어 조언으로 전환.

## 3. 쉬운 말 치환 가이드 (예시 사전)

계산은 그대로, 표현만 아래처럼. (AI가 문맥에 맞게 변주하되 이 결을 유지)

| 명리 용어 | 사용자에게 보여줄 쉬운 말 (괄호 병기 예) |
|---|---|
| 용신 | 나에게 꼭 필요한 기운(용신) |
| 희신 | 도움이 되는 기운 |
| 기신 | 조심하면 좋은 기운 |
| 일간 | 나의 본바탕 |
| 일지 합/육합/삼합 | 서로 잘 맞고 끌리는 기운(합) |
| 충 | 부딪히기 쉬운 기운 → "가끔 의견이 엇갈릴 수 있어요" |
| 형/파/해 | 작은 마찰 → "사소하게 삐걱일 수 있어요" |
| 상생 | 서로 북돋아주는 관계 |
| 상극 | 서로 힘겨루기가 될 수 있는 관계 |
| 조후 | 사주의 따뜻함·차가움 균형(온도) |
| 공망 | 비어 있는 자리 → 과하게 겁주지 말 것 |
| 식신·상관 | 표현력·재능 |
| 재성 | 재물·활동의 기운 |
| 관성 | 책임·자기관리의 기운 |
| 인성 | 배움·마음의 안정 |
| 원진살 | (용어 비노출) "이유 없이 살짝 엇갈릴 수 있으니 서로 먼저 물어봐 주세요" |

## 4. 멘트 생성 방식 — "방식 3" (뼈대 고정 + AI 변주)

- 코드/프롬프트로 고정하는 안전장치:
  - 금지: 명리 전문용어 과다 노출, 무서운 용어 노출, 불행·이별·바람기 단정, 겁주기.
  - 필수: 쉬운 말 우선 + 핵심 용어만 괄호, 따뜻한 마무리, 존댓말.
- 그 틀 안에서 AI가 사주·상황에 맞게 매번 자연스럽게 쓴다.
- 각 화면 프롬프트에 이 문서의 2·3항 요약과 본보기 문장을 포함시킨다.

## 5. 적용 대상 프롬프트(점진 반영)

아래 화면의 AI 프롬프트에 본 원칙을 공통 전제로 추가한다(작업 시 하나씩):
- 사주 기본 분석 (result/AiAnalysis 계열)
- 이름 풀이·개명 (/api/naming)
- 궁합·부부·택일 (useCoupleResult buildPrompt) — 궁합 설계서와 함께 적용
- 오늘의 운세 (daily_fortune 생성)
- 기타 통변 텍스트 생성 지점 전부

구현은 계산과 무관하므로 위험이 낮으나, 화면마다 프롬프트가 달라
한 번에 하나씩 반영·확인한다.
